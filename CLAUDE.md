# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`tuna.nvim` is a Neovim plugin for competitive programming, written in Lua. It is a ground-up rewrite/successor to [competitest.nvim](https://github.com/xeluxee/competitest.nvim), which is no longer actively maintained. The plugin integrates with [Competitive Companion](https://github.com/jmerle/competitive-companion) (a browser extension) to download problems/contests, manages testcases on disk, and compiles/runs solutions against them.

## Goals

- Written in idiomatic, modern Lua ‚Äî prefer current Neovim APIs (`vim.uv`, `vim.system`, etc.) over legacy/deprecated patterns.
- Prioritize speed: async I/O and process spawning wherever possible, minimal startup overhead.
- Don't just port competitest.nvim's design 1:1 ‚Äî research current Neovim plugin best practices (structure, config patterns, testing, docs) and apply them, improving on the original where it makes sense.
- Document changes and improvements with respect to competitest.nvim inside CHANGES.md, this will help to write the final README.md at the end of the project.

## Architecture

All Lua lives under `lua/tuna/`. The entry point is `lua/tuna/init.lua`, which calls `config.setup()` and registers the single `:Tuna <subcommand>` user command.

Module responsibilities:

- **`init.lua`** ‚Äî plugin setup, registers `:Tuna` command with tab-completion (plus a `cnoreabbrev` so a lowercase `:tuna ‚Ä¶` expands to `:Tuna`); highlight groups, `VimResized` resize (widgets + runner UIs + `stress.resize_all()`)
- **`config.lua`** ‚Äî layers defaults ‚Üí user `setup()` opts (`current_setup`) ‚Üí per-directory local config (`.tuna.lua`, found by walking up the tree from a buffer's file). `get_buffer_config(bufnr)` resolves + caches per-buffer config. `update_config_table` replaces per-language command `args` instead of index-merging them. Full schema covers compile/run, testcase storage (`testcases_storage` enum), receive, and UI options. `config.options` is a compat alias for `current_setup` read by the not-yet-ported runner
- **`commands.lua`** ‚Äî the `:Tuna <subcommand> [args‚Ä¶]` surface. `execute(fargs)` dispatches; `complete` gives context-aware tab-completion (subcommand names, then per-subcommand args for `run`/`convert`/`receive`/`scaffold`/`checker`). Handlers: `add_testcase`/`edit_testcase [n]`/`delete_testcase [n]` (wire the `widgets` editor/picker), `convert <files|single_file|directory>` (loads via auto-detect, clears all backends, rewrites to the target), `run [normal|all|stress|interactive] [args‚Ä¶]` ‚Äî the **run-mode dispatcher**: a leading mode keyword switches the buffer's active mode (via `tools.set_mode`, marking it explicit) and is consumed; otherwise the effective mode comes from `tools.resolve_mode` (an explicit prior choice if still usable, else auto-detected from the sibling helper files present). For `interactive` a further leading source keyword (`live|feed|interactor`) is passed through to `interactive.run` (remembered per buffer). `M.dispatch_mode` routes to `run_testcases`/`multi`/`stress`/`interactive` and records `M.last_mode[bufnr]`, which `show_results_ui`/`show_ui` use to re-open the right UI (stress or interactive keep a live `active[bufnr]` runner; else the normal runner). Also `run_no_compile [n‚Ä¶]`/`show_ui` (per-buffer `M.runners`, cleared on `BufUnload`), `checker [on|off|toggle]` (per-buffer special-judge toggle; drops the cached runner), `compare <exact|squish|float [tol]|default>` (per-buffer runtime override of `output_compare_method`, mirroring the checker toggle ‚Äî `float`‚Üí`{ "float", tol = tol or 1e-6 }`, `default` clears back to config; stored via `tools.set_compare`, drops the cached runner), `receive <testcases|problem|contest|persistently|status|stop>`, `scaffold <checker|generator|brute|interactor> [ext]`, `menu` (bare `:Tuna` opens the mode-switcher menu). The menu also carries a **Compare** entry that cycles `default ‚Üí exact ‚Üí squish ‚Üí float` (`M.cycle_compare`)
- **`receive.lua`** ‚Äî owns the Competitive Companion integration end-to-end (absorbed the old `http.lua`). Pipeline `Receiver` (TCP listener, default port `companion_port` 27121) ‚Üí `TasksCollector` (groups tasks by `batch.id`/`batch.size`) ‚Üí `BatchesSerialProcessor` (runs one batch handler at a time). `start_receiving(mode, ‚Ä¶)` supports modes `testcases`/`problem`/`contest`/`persistently`; storage helpers expand receive modifiers (`$(JUDGE)`, `$(CONTEST)`, `$(PROBLEM)`, ‚Ä¶) ‚Äî `$(JUDGE)`/`$(CONTEST)` come from `judges.parse` (see below) ‚Äî apply templates, prompt for paths via `widgets.input`, and write source + testcases through the `testcases` backends. `eval_receive_modifiers`/`eval_path` take the resolved `cfg` so per-directory `judge_parsers` and `date_format` apply. Exposes `status()`/`is_receiving()`/`mode()` for the lualine component and `show_status()`/`stop_receiving()`
- **`judges.lua`** (Phase 3, W3) ‚Äî turns Competitive Companion's `task.group` (`"Judge - Contest"`) into a tidy `judge` + `contest` for `$(JUDGE)`/`$(CONTEST)`. `M.parse(task, judge_parsers)` splits the group (lowercased), then applies the effective **parser** resolved as: `config.judge_parsers[judge]` (a user fn, or `false` to disable normalizing) ‚Üí `M.builtin[judge]` (shipped Codeforces/AtCoder normalizers, ported from the old inline hack in `receive.lua`) ‚Üí `config.judge_parsers["*"]` (user catch-all). A parser gets `{ judge, contest, group, task }` and returns `{ judge?, contest? }` overrides (nil fields keep the parsed values); it's `pcall`-guarded so a broken parser warns and falls back to the raw contest. Users add judges (CodeChef/USACO/‚Ä¶) or override/disable built-ins purely via `config.judge_parsers`, with no core edits
- **`testcases.lua`** ‚Äî reads/writes testcases via three interchangeable backends (`files`, `single_file`, `directory`) sharing a 0-based `{ [n] = { input, output } }` table. Pure `load`/`write` per backend; `buf_*` wrappers derive paths from buffer config; module-level `buf_get_testcases`/`buf_write_testcases`/`buf_save_testcase`/`buf_delete_testcase` dispatch to the configured backend with auto-detect fallback; `buf_clear` per backend supports `convert`. In `files` storage `testcases_input_file_format`/`testcases_output_file_format` may each be an **ordered list** of formats (or a single string): on load they're tried in order and the **first that discovers any testcase wins** (no merging ‚Üí two problems sharing a folder never cross-contaminate); the **first entry is canonical** (what `write` uses). The default is `{ "$(FNOEXT)_input$(TCNUM).txt", "input$(TCNUM).txt", "in.txt" }` (output analog `‚Ä¶, "out.txt"`) ‚Äî source-named first, then a shared un-prefixed `input<N>.txt` so any solution in a folder (e.g. `run all`, or CC-downloaded testcases from a differently-named source) discovers testcases it didn't write, then a **numberless** `in.txt`/`out.txt`. A format without `$(TCNUM)` names a **single** testcase (index 0) ‚Äî via `make_matcher`, which maps an exact filename match to `0` instead of capturing a number. A testcase may have **only an input or only an output**: an output with no matching input still runs (the solution is fed empty stdin), so an `out.txt`-only folder is runnable. `files.buf_clear` deletes every file matching **any** configured format (numberless included) so `convert` cleans up fallback-named testcases too. `add`/`load_first` are deprecated shims for the not-yet-ported runner/commands
- **`runner/core.lua`** ‚Äî `RunnerCore`, the shared base every run mode subclasses (normal, stress, interactive; run-all later). It owns the half of a runner that `runner_ui` drives ‚Äî the `tcdata` rows, the UI plumbing (`update_ui`/`show_ui`/`resize_ui`/`delete_ui`), `judge_label`, kill helpers ‚Äî plus one reusable **`execute_process(tcindex, cmd, dir, opts, on_done)`** that spawns a `vim.system` child with its own timeout timer (labels TIMEOUT vs KILLED/SIG/RET), and on a clean exit judges via `checker.judge` (unless `opts.judge == false`, the Compile row) setting the `judging` flag for async external checkers. Two seams let a mode specialize the UI without the UI knowing: `pane_content(tc, name)` (what each detail pane shows; return `core.SKIP` to leave a pane untouched ‚Äî interactive's editable Input pane) and `on_ui_shown(ui)` (augment the freshly-built UI). Inheritance is plain metatable single-dispatch via `M.extend()`.
- **`runner/init.lua`** (`require("tuna.runner")`) ‚Äî the normal run mode plus the shared **resolver** `M.new(bufnr)` (evaluates the compile & run commands, dirs, and checker) that stress/interactive/multi all reuse. `TCRunner` subclasses `RunnerCore`; `run_testcases(tctbl, do_compile)` builds `tcdata` (a compile pseudo-testcase at index 1 when compiling, then one per testcase) and runs them through `execute_process` up to `multiple_testing` in parallel. The compile step gates the rest (testcases run only if it exits 0); `check_complete` waits on all `running`/`judging` rows. `run_single(tcindex)` re-runs one; re-run all via `run_testcases(nil)`. `display_results()` is a fallback float used only if the UI can't be created
- **`compare.lua`** ‚Äî output comparison: `compare_output(output, expected, method)` with builtin `exact`/`squish`/`float` methods and custom-function support; returns `true`/`false`/`nil`. `float` (a `{ "float", tol = 1e-6 }` table ‚Äî builtin name in `[1]`, options in the table) compares token-wise, accepting numeric tokens within `tol` absolute *or* relative error while non-numeric tokens must match exactly. `method_name(method)` renders a method for the results-UI status pane (e.g. `float, tol=1e-06`, `custom`)
- **`tools.lua`** (Phase 3) ‚Äî helper-program abstraction + per-buffer run state. `find(dir, role, cfg)` discovers a sibling helper source file by filename convention (`checker.*`/`gen.*`/`brute.*`/`interactor.*`, names from `tool_names`); `program(path, cfg)`/`checker_spec(path, cfg)` resolve a path into a runnable spec (`{ exec, args, compile?, compile_dir?, cwd }`) using the file's own filetype‚Üícompile/run commands; `prepare(spec, cb)` compiles the spec and caches the result in a **persistent, session-wide cache keyed by source path + mtime + compile command** (queuing concurrent callers so parallel judges don't recompile/race), so repeated runs reuse an unchanged build and only recompile what actually changed. Also holds the per-buffer run state keyed by file path: `get_mode`/`set_mode` (`normal`/`all`/`stress`/`interactive`; `set_mode` marks the mode as an explicit user choice), `detect_mode(dir, cfg)` (auto-pick from files present ‚Äî `interactor.*` ‚áí interactive, `gen.*`+`brute.*` ‚áí stress, else normal) and `resolve_mode(path, dir, cfg)` (explicit choice if its required files still exist, else `detect_mode`), plus `checker_enabled`/`set_checker`, `get_source`/`set_source`, and `get_compare`/`set_compare` (the per-buffer `:Tuna compare` override ‚Äî nil means use `config.output_compare_method`). Every runner reads the effective method via `RunnerCore:effective_compare()` (`self.compare_method` override else config), and `runner.new`/the stress/interactive/multi constructors seed `compare_method` from `tools.get_compare`
- **`checker.lua`** (Phase 3) ‚Äî verdict layer. `judge(tc, checker, compare_method, callback)`: `"builtin"` delegates to `compare.lua`; an external testlib-style checker is invoked as `checker <input> <output> <answer>` (placeholders `$(INPUT)`/`$(OUTPUT)`/`$(ANSWER)`), exit 0 = correct, stderr/stdout = `tc.checker_message`. The external checker is usually a source file ‚Äî `checker.judge` compiles it on first use via `tools.prepare` (cached) and spawns it with `checker.cwd`. Async (callback-based) because the external path spawns processes. `runner.new()` resolves the checker: the per-buffer toggle can force `"builtin"`, else a discovered `checker.*` / a config path (source or binary) / a `{ exec, args }` prebuilt
- **`stress.lua`** (Phase 3) ‚Äî stress testing (`:Tuna run stress [count]`). Reuses `runner.new()` for the solution's resolved compile/run commands + checker; discovers `gen.*`/`brute.*` via `tools.find` (config `stress.generator`/`stress.reference` override) and compiles them via `tools.prepare`; then loops: `generator(seed)` ‚Üí input, runs the solution and the `reference` on it, judges via `checker.judge` (sol output vs reference output ‚Äî a custom checker works here too). Drives its **own `runner_ui`** via a `StressRunner` (a `RunnerCore` subclass ‚Äî it inherits the UI plumbing and `execute_process`, overrides `kill_*` to "stop the search", and adds `status_tail`): the UI opens immediately (with the solution's compile step as the first `Compile` testcase row, so its warnings are viewable) and, as soon as the solution is built, re-runs the existing testcases **in parallel with** the generator/reference compiling (generation waits for both). Each counterexample (input + reference answer as expected) is appended as a new saved testcase and a UI row (shown as `saved` in the time column) ‚Äî but only if its input differs from every testcase already present (dedup). The search stops when either threshold is hit ‚Äî `saves_per_run` saved this run, or `max_saved` total testcases ‚Äî surfaced live in the status pane (iter/saved/thresholds). A compile failure is shown in the UI (the "Run" pane's compile line + a message float via `runner_ui:show_message`), not as an error notification. `M.active[bufnr]` tracks live runners (`resize_all()` rebuilds on `VimResized`). Config under `stress = { generator, reference, count, seed_arg, saves_per_run, max_saved }`
- **`interactive.lua`** (Phase 3) ‚Äî interactive problems (`:Tuna run interactive [live|feed|interactor] [n‚Ä¶]`). An `InteractiveRunner` (`RunnerCore` subclass with its own `runner_ui`, `M.active[bufnr]`, `resize_all`) runs one testcase session at a time, with three input **sources** (chosen by the leading keyword, remembered per buffer via `tools.get_source`/`set_source`; default `interactor` when an `interactor.*`/config interactor exists, else `live`): **live** ‚Äî you are the other side: `on_ui_shown` makes the Input pane editable and maps `<CR>` to send the current line to the solution (`live_send`), its stdout streams into Output; no auto-verdict (DONE). **feed** ‚Äî the testcase input plays the other side one line per turn (prime the first line, then send the next each time the solution emits a line); judged against expected output if present. **interactor** ‚Äî a discovered `interactor.*` (or `interactive.interactor`, compiled via `tools.prepare`) cross-wired to the solution over `vim.uv.spawn` pipes (the exchange teed into the transcript, interactor stderr into Errors), verdict = interactor exit code, receiving the testcase input/answer via `$(INPUT)`/`$(ANSWER)`. All reuse `runner.new()` for solution resolution; the solution compile is a `Compile` row like the other runners
- **`multi.lua`** (Phase 3) ‚Äî multiple solution *versions* (`:Tuna run all`). Runs every runnable sibling source file ‚Äî **of any language**, each with its *own* filetype's compile/run commands (so a C++ and a Python attempt run together) ‚Äî against the shared testcases, **excluding helper files** (`checker.*`/`gen.*`/`brute.*`/`interactor.*`), compiling each and judging via the `checker` (resolved once for the dir via `tools`); reports a per-solution `correct/total` summary. Commands are resolved per file with `utils.eval_string` (no open buffer). The problem directory is the invoking buffer's parent when it's a real file, else the **cwd** ‚Äî so `:Tuna run all` works from an unnamed/scratch buffer (just `nvim` into a folder); testcases then come from the invoking buffer, or (no current file) are auto-discovered in the dir anchored on the first solution (so shared `input<N>.txt` are found). Empty discovery is a single **WARN** naming the dir searched, not a grey one-liner. Distinct from multiple-*answer* problems, which the `checker` already handles. A `MultiRunner` (`RunnerCore` subclass, `M.active[bufnr]`, `resize_all`) drives the shared `runner_ui` as a **flattened matrix**: a solution *header* row (via the `row_label` seam ‚Äî name + a live `correct/total`) above its indented per-testcase rows. It compiles every solution first (in parallel), then runs all cases across all solutions through a shared pool of `multiple_testing` concurrent `execute_process` calls (headers update live as cases land). Selecting a testcase row shows that run in the detail panes; selecting a solution row shows its per-testcase summary (Output) and compile output (Errors); a compile failure is a `CE` row (cases marked `‚Äî`), not an error notify. `run_single` re-runs one case (or a whole solution from its header row)
- **`scaffold.lua`** (Phase 3) ‚Äî `:Tuna scaffold <checker|generator|brute|interactor> [ext]`: writes a dependency-free starter file (in the solution's language by default, or `[ext]`) into the problem directory and opens it. Built-in C++/Python templates; base filename and per-language templates (`{ [ext] = path }`) overridable via `config.scaffold.{files,templates}`
- **`checker`/`stress`/`interactive`/`multi`/`scaffold`** are also reachable through the **mode-switcher menu** (`widgets.menu`): bare `:Tuna` (or `:Tuna menu`) opens a native chooser that switches the buffer's run mode (normal / run-all / stress / interactive ‚Äî what a later `:Tuna run` repeats), toggles the checker, shows the results UI, or scaffolds a helper (precursor to the W5 dashboard)
- **`widgets.lua`** ‚Äî native floating-window widgets (no nui): `input` single-line prompt, `editor` two-pane testcase editor (`:w`/`:wq` save via `BufWriteCmd`), `picker` testcase chooser, `menu` generic single-choice chooser (Phase 3 mode switcher). Each is a module-level singleton so `resize_widgets()` can rebuild visible ones on `VimResized`. Config under `editor_ui`/`picker_ui`/`floating_border`
- **`runner_ui/`** ‚Äî native (no nui) results UI driving any `RunnerCore` (normal/stress/interactive/run-all). `init.lua` (`RunnerUI`) owns a **"Run" status pane** (`st`) above a testcase selector + four detail panes (stdout/expected/stdin/stderr), wires the keymaps (run again/all, kill/kill all, view-in-viewer, toggle diff, close, and `switch_window` directional pane focus), tracks the selected testcase via `CursorMoved`, and re-renders from `runner.tcdata` on the `update_windows`/`update_details` flags. It stays mode-agnostic through three runner seams: it asks `runner:pane_content(tc, name)` what each detail pane should show (skipping panes that return `core.SKIP`), `runner:row_label(tc)` for the selector's left column (run-all relabels solution/testcase rows), and calls `runner:on_ui_shown(self)` after building the windows. The `st` pane (`status_lines()`) shows the run mode, the verdict source, and any runner tail (`status_tail`, e.g. stress iteration/save counters), as colon-aligned `label: value` rows; its row count is stable per runner and sizes the pane at build time. The **compile step is a testcase row** (`Compile`, at `tcdata[1]`) in the selector ‚Äî as in competitest ‚Äî so its warnings are viewable in the detail panes and a failure auto-pops the viewer on stderr. `switch_window` (default `<M-hjkl>`) moves focus between panes by window geometry, so it works for the floating popup interface too where core `<C-w>hjkl` can't cross floats. `popup.lua` and `split.lua` are interchangeable layout interfaces (selected by `runner_ui.interface`) sharing one recursive `{ ratio, child }` layout engine ‚Äî `popup` tiles floats, `split` builds native splits. The viewer is a large float that borrows a detail pane's buffer; diff view runs `diffthis` on the output/expected panes. State lives in `tcdata`, so close tears down and show rebuilds; `resize_ui()` rebuilds on `VimResized`
- **`utils.lua`** ‚Äî `notify`; `set_border_highlight` (remaps floats' `FloatBorder` to a derived `TunaFloatBorder` that keeps the configured foreground but takes the editor's `Normal` background, so a rounded border's corners blend into the editor and the `NormalFloat` fill appears to follow the rounding); modifier engine (`format_modifiers` state-machine parser, `file_format_modifiers`, `eval_string`, `buf_eval_string`); filesystem helpers (`file_exists`, `directory_exists`, `ensure_directory`, `read_file`, `write_file`, `delete_file`, `normalize_path`); `get_ui_size`

### Key data flow

1. User runs `:Tuna receive problem` ‚Üí `commands.receive` ‚Üí `receive.start_receiving("problem", ‚Ä¶)` opens the TCP listener
2. Competitive Companion POSTs JSON ‚Üí `Receiver` decodes each task ‚Üí `TasksCollector` waits for the full batch ‚Üí `BatchesSerialProcessor` runs the mode's handler ‚Üí source + testcase files written to disk (and the source opened)
3. User runs `:Tuna run` ‚Üí `runner.new(bufnr)` builds a `TCRunner` ‚Üí `commands` loads testcases via `testcases.buf_get_testcases` and calls `runner:run_testcases(tctbl, true)` ‚Üí compile (if any) then parallel `vim.system` runs, each compared against expected output; results shown in the (temporary) results float

### Command modifiers (template variables)

Used in `compile_command` / `run_command` args: `$(FNAME)`, `$(FNOEXT)`, `$(FEXT)`, `$(FABSPATH)`, `$(ABSDIR)`. Expanded by `utils.eval_string` / `utils.buf_eval_string`.

### Testcase directory layout

The on-disk layout is chosen by `testcases_storage` (`files` | `single_file` | `directory`) and the matching format options ‚Äî see `testcases.lua` and DIFFERENCES.md. The default `files` mode keeps an input/output pair beside the source (e.g. `sol_input0.txt` / `sol_output0.txt`) and additionally discovers shared, un-prefixed pairs (`input0.txt` / `output0.txt`) and a numberless single-testcase `in.txt` / `out.txt` as fallbacks, so any solution in the folder can run testcases it didn't create (an `out.txt` with no input runs against empty stdin); `directory` mode uses one sub-directory per testcase (e.g. `tests/0/input.txt` + `output.txt`) and is inherently source-agnostic. The runner loads all testcases via `testcases.buf_get_testcases` and runs them together (parallelised), not just the first.

## Development

This is a pure Lua Neovim plugin ‚Äî there is no build step, test suite, or package manager. Load it in Neovim via your plugin manager pointing at this repo. Local per-project config can be placed in `tuna.lua` or `.tuna.lua` at the project root.
