This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`tuna.nvim` is a Neovim plugin for competitive programming, written in Lua. It is a ground-up rewrite/successor to [competitest.nvim](https://github.com/xeluxee/competitest.nvim), which is no longer actively maintained. The plugin integrates with [Competitive Companion](https://github.com/jmerle/competitive-companion) (a browser extension) to download problems/contests, manages testcases on disk, and compiles/runs solutions against them.

## Goals

- Written in idiomatic, modern Lua — prefer current Neovim APIs (`vim.uv`, `vim.system`, etc.) over legacy/deprecated patterns.
- Prioritize speed: async I/O and process spawning wherever possible, minimal startup overhead.
- Don't just port competitest.nvim's design 1:1 — research current Neovim plugin best practices (structure, config patterns, testing, docs) and apply them, improving on the original where it makes sense.
- Document changes and improvements with respect to competitest.nvim inside CHANGES.md, this will help to write the final README.md at the end of the project.

## Architecture

All Lua lives under `lua/tuna/`. The entry point is `lua/tuna/init.lua`, which calls `config.setup()` and registers the single `:Tuna <subcommand>` user command.

Module responsibilities:

- **`init.lua`** — plugin setup, registers `:Tuna` command with tab-completion (plus a `cnoreabbrev` so a lowercase `:tuna …` expands to `:Tuna`); highlight groups, `VimResized` resize (widgets + runner UIs + `stress.resize_all()`)
- **`config.lua`** — layers defaults → user `setup()` opts (`current_setup`) → per-directory local config (`.tuna.lua`, found by walking up the tree from a buffer's file). `get_buffer_config(bufnr)` resolves + caches per-buffer config. `update_config_table` replaces per-language command `args` instead of index-merging them. Full schema covers compile/run, testcase storage (`testcases_storage` enum), receive, and UI options. `config.options` is a compat alias for `current_setup` read by the not-yet-ported runner
- **`commands.lua`** — the `:Tuna <subcommand> [args…]` surface. `execute(fargs)` dispatches; `complete` gives context-aware tab-completion (subcommand names, then per-subcommand args for `run`/`convert`/`receive`/`scaffold`/`checker`). Handlers: `add_testcase`/`edit_testcase [n]`/`delete_testcase [n]` (wire the `widgets` editor/picker), `convert <files|single_file|directory>` (loads via auto-detect, clears all backends, rewrites to the target), `run [normal|all|stress|interactive] [args…]` — the **run-mode dispatcher**: a leading mode keyword switches the buffer's active mode (via `tools.set_mode`, marking it explicit) and is consumed; otherwise the effective mode comes from `tools.resolve_mode` (an explicit prior choice if still usable, else auto-detected from the sibling helper files present). `M.dispatch_mode` routes to `run_testcases`/`multi`/`stress`/`interactive`. Also `run_no_compile [n…]`/`show_ui` (per-buffer `M.runners`, cleared on `BufUnload`), `checker [on|off|toggle]` (per-buffer special-judge toggle; drops the cached runner), `receive <testcases|problem|contest|persistently|status|stop>`, `scaffold <checker|generator|brute|interactor> [ext]`, `menu` (bare `:Tuna` opens the mode-switcher menu)
- **`receive.lua`** — owns the Competitive Companion integration end-to-end (absorbed the old `http.lua`). Pipeline `Receiver` (TCP listener, default port `companion_port` 27121) → `TasksCollector` (groups tasks by `batch.id`/`batch.size`) → `BatchesSerialProcessor` (runs one batch handler at a time). `start_receiving(mode, …)` supports modes `testcases`/`problem`/`contest`/`persistently`; storage helpers expand receive modifiers (`$(JUDGE)`, `$(CONTEST)`, `$(PROBLEM)`, …), apply templates, prompt for paths via `widgets.input`, and write source + testcases through the `testcases` backends. Exposes `status()`/`is_receiving()`/`mode()` for the lualine component and `show_status()`/`stop_receiving()`
- **`testcases.lua`** — reads/writes testcases via three interchangeable backends (`files`, `single_file`, `directory`) sharing a 0-based `{ [n] = { input, output } }` table. Pure `load`/`write` per backend; `buf_*` wrappers derive paths from buffer config; module-level `buf_get_testcases`/`buf_write_testcases`/`buf_save_testcase`/`buf_delete_testcase` dispatch to the configured backend with auto-detect fallback; `buf_clear` per backend supports `convert`. `add`/`load_first` are deprecated shims for the not-yet-ported runner/commands
- **`runner.lua`** — `TCRunner` execution engine. `new(bufnr)` resolves/evaluates the compile & run commands; `run_testcases(tctbl, do_compile)` builds `tcdata` (a compile pseudo-testcase at index 1 when compiling, then one entry per testcase) and runs them as `vim.system` processes, up to `multiple_testing` in parallel. Each process has its own timeout timer (labels TIMEOUT vs KILLED/SIG/RET) and result status (CORRECT/WRONG/DONE via `checker.judge`). A clean-exit verdict may be async (external checker), so each testcase carries a `judging` flag that `check_complete` waits on. The compile step gates the rest: testcases run only if it exits 0. `kill_process`/`kill_all_processes`; `run_single(tcindex)` re-runs one; re-run all via `run_testcases(nil)`. `show_ui()` lazily creates the `runner_ui` and drives it through the `update_ui` hook; `display_results()` is a fallback float used only if the UI can't be created
- **`compare.lua`** — output comparison: `compare_output(output, expected, method)` with builtin `exact`/`squish` methods and custom-function support; returns `true`/`false`/`nil`
- **`tools.lua`** (Phase 3) — helper-program abstraction + per-buffer run state. `find(dir, role, cfg)` discovers a sibling helper source file by filename convention (`checker.*`/`gen.*`/`brute.*`/`interactor.*`, names from `tool_names`); `program(path, cfg)`/`checker_spec(path, cfg)` resolve a path into a runnable spec (`{ exec, args, compile?, compile_dir?, cwd }`) using the file's own filetype→compile/run commands; `prepare(spec, cb)` compiles the spec and caches the result in a **persistent, session-wide cache keyed by source path + mtime + compile command** (queuing concurrent callers so parallel judges don't recompile/race), so repeated runs reuse an unchanged build and only recompile what actually changed. Also holds the per-buffer run state keyed by file path: `get_mode`/`set_mode` (`normal`/`all`/`stress`/`interactive`; `set_mode` marks the mode as an explicit user choice), `detect_mode(dir, cfg)` (auto-pick from files present — `interactor.*` ⇒ interactive, `gen.*`+`brute.*` ⇒ stress, else normal) and `resolve_mode(path, dir, cfg)` (explicit choice if its required files still exist, else `detect_mode`), plus `checker_enabled`/`set_checker`
- **`checker.lua`** (Phase 3) — verdict layer. `judge(tc, checker, compare_method, callback)`: `"builtin"` delegates to `compare.lua`; a Lua-function checker runs in-process; an external testlib-style checker is invoked as `checker <input> <output> <answer>` (placeholders `$(INPUT)`/`$(OUTPUT)`/`$(ANSWER)`), exit 0 = correct, stderr/stdout = `tc.checker_message`. The external checker is usually a source file — `checker.judge` compiles it on first use via `tools.prepare` (cached) and spawns it with `checker.cwd`. Async (callback-based) because the external path spawns processes. `runner.new()` resolves the checker: the per-buffer toggle can force `"builtin"`, else a Lua function / a discovered `checker.*` / a config path (source or binary) / a `{ exec, args }` prebuilt
- **`stress.lua`** (Phase 3) — stress testing (`:Tuna run stress [count]`). Reuses `runner.new()` for the solution's resolved compile/run commands + checker; discovers `gen.*`/`brute.*` via `tools.find` (config `stress.generator`/`stress.reference` override) and compiles them via `tools.prepare`; then loops: `generator(seed)` → input, runs the solution and the `reference` on it, judges via `checker.judge` (sol output vs reference output — a custom checker works here too). Drives its **own `runner_ui`** via a `StressRunner` (a `TCRunner`-shaped object exposing `tcdata`/`mode`/`judge_label`/`compile_info`/`status_tail`/`kill_*`/`run_single`/`run_testcases`): the UI opens immediately (with the solution's compile step as the first `Compile` testcase row, so its warnings are viewable) and, as soon as the solution is built, re-runs the existing testcases **in parallel with** the generator/reference compiling (generation waits for both). Each counterexample (input + reference answer as expected) is appended as a new saved testcase and a UI row (shown as `saved` in the time column) — but only if its input differs from every testcase already present (dedup). The search stops when either threshold is hit — `saves_per_run` saved this run, or `max_saved` total testcases — surfaced live in the status pane (iter/saved/thresholds). A compile failure is shown in the UI (the "Run" pane's compile line + a message float via `runner_ui:show_message`), not as an error notification. `M.active[bufnr]` tracks live runners (`resize_all()` rebuilds on `VimResized`). Config under `stress = { generator, reference, count, seed_arg, saves_per_run, max_saved }`
- **`interactive.lua`** (Phase 3) — interactive problems (`:Tuna run interactive [n…]`). Discovers `interactor.*` via `tools.find` (config `interactive.interactor` overrides) and compiles it via `tools.prepare`; spawns the solution and interactor via `vim.uv.spawn` and cross-wires their stdio (sol.stdout→int.stdin, int.stdout→sol.stdin); the interactor is the verdict authority (exit 0 = CORRECT). The interactor receives the testcase input/answer through `$(INPUT)`/`$(ANSWER)` temp-file placeholders (appended to a discovered interactor's args). Per-testcase sequential, timeout-guarded; reuses `runner.new()` for solution resolution
- **`multi.lua`** (Phase 3) — multiple solution *versions* (`:Tuna run all`). Runs every sibling source file with the current file's extension (so they share its filetype/commands) against the shared testcases — **excluding helper files** (`checker.*`/`gen.*`/`brute.*`/`interactor.*`) — compiling each and judging via the `checker` (resolved once for the dir via `tools`); reports a per-solution `correct/total` summary. Commands are resolved per file with `utils.eval_string` (no open buffer). Distinct from multiple-*answer* problems, which the `checker` already handles
- **`scaffold.lua`** (Phase 3) — `:Tuna scaffold <checker|generator|brute|interactor> [ext]`: writes a dependency-free starter file (in the solution's language by default, or `[ext]`) into the problem directory and opens it. Built-in C++/Python templates; base filename and per-language templates (`{ [ext] = path }`) overridable via `config.scaffold.{files,templates}`
- **`checker`/`stress`/`interactive`/`multi`/`scaffold`** are also reachable through the **mode-switcher menu** (`widgets.menu`): bare `:Tuna` (or `:Tuna menu`) opens a native chooser that switches the buffer's run mode (normal / run-all / stress / interactive — what a later `:Tuna run` repeats), toggles the checker, shows the results UI, or scaffolds a helper (precursor to the W5 dashboard)
- **`widgets.lua`** — native floating-window widgets (no nui): `input` single-line prompt, `editor` two-pane testcase editor (`:w`/`:wq` save via `BufWriteCmd`), `picker` testcase chooser, `menu` generic single-choice chooser (Phase 3 mode switcher). Each is a module-level singleton so `resize_widgets()` can rebuild visible ones on `VimResized`. Config under `editor_ui`/`picker_ui`/`floating_border`
- **`runner_ui/`** — native (no nui) results UI driving a `TCRunner`. `init.lua` (`RunnerUI`) owns a **"Run" status pane** (`st`) above a testcase selector + four detail panes (stdout/expected/stdin/stderr), wires the keymaps (run again/all, kill/kill all, view-in-viewer, toggle diff, close, and `switch_window` directional pane focus), tracks the selected testcase via `CursorMoved`, and re-renders from `runner.tcdata` on the `update_windows`/`update_details` flags. The `st` pane (`status_lines()`) shows the run mode, the verdict source, and any runner tail (`status_tail`, e.g. stress iteration/save counters), as colon-aligned `label: value` rows; its row count is stable per runner and sizes the pane at build time. The **compile step is a testcase row** (`Compile`, at `tcdata[1]`) in the selector — as in competitest — so its warnings are viewable in the detail panes and a failure auto-pops the viewer on stderr. `switch_window` (default `<M-hjkl>`) moves focus between panes by window geometry, so it works for the floating popup interface too where core `<C-w>hjkl` can't cross floats. `popup.lua` and `split.lua` are interchangeable layout interfaces (selected by `runner_ui.interface`) sharing one recursive `{ ratio, child }` layout engine — `popup` tiles floats, `split` builds native splits. The viewer is a large float that borrows a detail pane's buffer; diff view runs `diffthis` on the output/expected panes. State lives in `tcdata`, so close tears down and show rebuilds; `resize_ui()` rebuilds on `VimResized`
- **`utils.lua`** — `notify`; `set_border_highlight` (remaps floats' `FloatBorder` to a derived `TunaFloatBorder` that keeps the configured foreground but takes the editor's `Normal` background, so a rounded border's corners blend into the editor and the `NormalFloat` fill appears to follow the rounding); modifier engine (`format_modifiers` state-machine parser, `file_format_modifiers`, `eval_string`, `buf_eval_string`); filesystem helpers (`file_exists`, `directory_exists`, `ensure_directory`, `read_file`, `write_file`, `delete_file`, `normalize_path`); `get_ui_size`

### Key data flow

1. User runs `:Tuna receive problem` → `commands.receive` → `receive.start_receiving("problem", …)` opens the TCP listener
2. Competitive Companion POSTs JSON → `Receiver` decodes each task → `TasksCollector` waits for the full batch → `BatchesSerialProcessor` runs the mode's handler → source + testcase files written to disk (and the source opened)
3. User runs `:Tuna run` → `runner.new(bufnr)` builds a `TCRunner` → `commands` loads testcases via `testcases.buf_get_testcases` and calls `runner:run_testcases(tctbl, true)` → compile (if any) then parallel `vim.system` runs, each compared against expected output; results shown in the (temporary) results float

### Command modifiers (template variables)

Used in `compile_command` / `run_command` args: `$(FNAME)`, `$(FNOEXT)`, `$(FEXT)`, `$(FABSPATH)`, `$(ABSDIR)`. Expanded by `utils.eval_string` / `utils.buf_eval_string`.

### Testcase directory layout

The on-disk layout is chosen by `testcases_storage` (`files` | `single_file` | `directory`) and the matching format options — see `testcases.lua` and DIFFERENCES.md. The default `files` mode keeps an input/output pair beside the source (e.g. `sol_input0.txt` / `sol_output0.txt`); `directory` mode uses one sub-directory per testcase (e.g. `tests/0/input.txt` + `output.txt`). The runner loads all testcases via `testcases.buf_get_testcases` and runs them together (parallelised), not just the first.

## Development

This is a pure Lua Neovim plugin — there is no build step, test suite, or package manager. Load it in Neovim via your plugin manager pointing at this repo. Local per-project config can be placed in `tuna.lua` or `.tuna.lua` at the project root.

Optional lualine integration:

```lua
require("lualine").setup({ sections = { lualine_x = { require("tuna").lualine_component } } })
```
